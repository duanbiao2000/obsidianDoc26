下面是一个接近 **Google / Stripe / OpenAI 内部实践** 的 **L6 / L7 级技术文档优先级模型**。
重点不是“写哪些文档”，而是建立一种 **可持续扩展的知识决策系统** —— 即让文档成为工程杠杆，而不是记录行为。

---

# 一、L6/L7 文档观：文档 = 规模化系统

在高级工程/产品组织中，文档的目标不是解释代码，而是：

```text
降低系统的认知成本（Cognitive Cost）
```

L3–L5 写的是：

> 我做了什么

L6–L7 写的是：

> 如何让 1000 人不用再问同样的问题

---

# 二、核心模型：Documentation Value Pyramid

## [[文档价值金字塔]]（从高 → 低优先级）

```
        █ Strategic Decision Docs        (L7)
      █████ System Mental Models
    █████████ Common Failure Guides
  █████████████ Task Execution Docs
██████████████████ Getting Started
```

---

## ① Getting Started（最高 ROI）

### 目标

让新人 **30 分钟成功运行系统**

### 包含

- Quickstart
- 最小 runnable example
- 环境配置
- 5 分钟 demo

### 为什么优先级最高？

因为：

```text
每个新用户都会经历一次 onboarding
```

价值公式：

```
价值 = 新用户数量 × 首次失败成本
```

这是组织最大隐性损耗来源。

---

## ② Task Execution Docs（高频任务）

回答：

> “我现在要完成 X，该怎么做？”

例如：

- 如何调用 API
- 如何部署服务
- 如何写插件
- 如何计算 tokens
- 如何调试请求

特点：

- 搜索导向
- copy-paste 可运行
- 无理论铺垫

---

## ③ Common Failure Guides（L6 分水岭）

这是 **普通团队 vs 高成熟团队** 的核心区别。

回答：

> 为什么会失败？

典型内容：

- 常见错误原因
- Debug checklist
- Anti-pattern
- 性能陷阱
- rate limit 解决方案

优秀组织 40% 文档都在这里。

原因：

```text
用户失败次数 >> 成功次数
```

减少失败 = 最大效率提升。

---

## ④ System Mental Models（L6 层）

不是 HOW，而是：

> 系统是如何思考的？

例如：

- 请求生命周期图
- 数据流模型
- cache 机制
- agent execution flow
- consistency model

目标：

```text
减少错误抽象
```

当用户理解模型后：

- 不再依赖文档
- 能自主推理问题

这是 L6 标志。

---

## ⑤ Strategic Decision Docs（L7 层）

回答：

> 为什么系统被设计成这样？

包括：

- Architecture Decision Records (ADR)
- tradeoff 分析
- rejected alternatives
- scaling reasoning

作用：

- 防止组织重复讨论
- 新负责人快速继承上下文

这是 **组织记忆层**。

---

# 三、L6/L7 优先级评分模型（可操作）

定义：

```
Doc Priority Score =
(Frequency × Blast Radius × Time Saved)
÷ Cognitive Load to Write
```

---

## 三个核心维度

### 1️⃣ Frequency（出现频率）

多少人遇到？

|频率|权重|
|---|---|
|每天|5|
|每周|4|
|每月|3|
|偶尔|1|

---

### 2️⃣ Blast Radius（影响范围）

|影响|权重|
|---|---|
|阻塞开发|5|
|影响性能|4|
|影响体验|3|
|小优化|1|

---

### 3️⃣ Time Saved（节省时间）

平均节省：

- 5 分钟 → 1
- 30 分钟 → 3
- 半天 → 5

---

## 示例

|文档|Score|
|---|---|
|Token 计算|5×5×4 = 100|
|Emoji DB 优化|1×1×2 = 2|

优先级差距 = **50×**

# 四、L6 vs L7 的真正区别

|维度|L6|L7|
|---|---|---|
|关注点|团队效率|组织规模化|
|文档目标|减少重复问题|改变思考方式|
|时间尺度|当前系统|未来系统|
|输出|Playbook|Principles|

---

# 五、顶级团队的文档比例（真实分布）

高成熟组织通常：

|类型|占比|
|---|---|
|Getting Started|20%|
|Task Docs|25%|
|Failure Guides|30% ⭐|
|Mental Models|15%|
|Decision Docs|10%|

多数团队错误：

```
80% API reference
5% failure docs
```

---

# 六、一个 L7 级洞察（关键）

真正高级的文档不是：

> 回答问题

而是：

> 让问题不再被提出

即：

```
Documentation → Interface
```

当文档足够好时：

- Slack 问题下降
- onboarding 时间下降
- bug 类型收敛
- 架构一致性提高

文档开始像 API 一样工作。

---

# 七、可直接落地的执行顺序（建议）

如果你今天接管一个项目：

**按这个顺序写：**

1. Quickstart（必须）
2. Top 10 failures
3. Debug flowchart
4. Mental model diagram
5. ADR 历史

不要反过来。

---

# 八、L7 级最终原则（可作为团队标准）

```
Write docs for future scale, not present clarity.
Optimize for questions that will be asked 1000 times.
Reduce thinking cost, not explanation depth.
```
