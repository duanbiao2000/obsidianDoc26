## 底层需求识别
你想理解**数据团队不同角色如何协作完成一个端到端的业务场景**——从原始数据到最终的智能化决策支持。

---

## 实例场景：电商平台"用户流失预警系统"

### **业务目标**
降低高价值用户流失率，从被动响应到主动干预

---

## 工作流流转（按时间线展开）

### **Phase 1: 数据基础建设（Week 1-2）**
**数据架构师 + 数据工程师**

**架构师产出**：
```
数据分层设计：
├─ ODS层：原始业务日志（用户行为、订单、客服）
├─ DWD层：清洗后明细表
├─ DWS层：用户特征宽表（7日登录次数、30日购买金额等）
└─ ADS层：面向应用的聚合表
```

**工程师实现**：
- 用Airflow构建ETL管道：每日凌晨2点从MySQL/MongoDB抽取增量数据
- 在Snowflake建立维度建模（用户维度表、商品维度表、事实表）
- 关键产出：`dws_user_behavior_feature` 表，包含120+特征字段

**踩坑点**：处理用户跨设备行为合并问题（同一用户iOS+Android的ID mapping）

---

### **Phase 2: 探索性分析（Week 3）**
**数据分析师**

**任务**：找到流失用户的共性特征

**工作内容**：
```sql
-- 在 dws_user_behavior_feature 基础上分析
SELECT 
  CASE WHEN last_login_days > 30 THEN '流失' ELSE '活跃' END AS user_status,
  AVG(purchase_frequency_30d) AS avg_purchase,
  AVG(customer_service_contact_count) AS avg_cs_contact
FROM dws_user_behavior_feature
GROUP BY 1
```

**发现洞察**：
- 流失用户特征：30天内购买频次<2、客服接触>3次、优惠券使用率<10%
- 产出：《高价值用户流失分析报告.pptx》提交给业务方

---

### **Phase 3: 报表自动化（Week 4）**
**BI开发者**

**问题**：业务方每周都要这份报告，分析师手动跑SQL不可持续

**解决方案**：
- 在Tableau搭建实时监控Dashboard
  - KPI卡片：本周流失用户数、环比变化
  - 漏斗图：注册→首购→复购→流失各环节转化率
  - 细分分析：按用户等级、品类偏好下钻
- 设置自动邮件：每周一早8点发送至运营团队

**技术选型依据**：
- 用Tableau而非自研：业务团队已熟悉其交互逻辑，培训成本低
- 数据源直连Snowflake而非导出CSV：保证实时性

---

### **Phase 4: 预测模型开发（Week 5-7）**
**数据科学家**

**升级需求**：从"事后统计"到"提前7天预测谁会流失"

**建模流程**：
```python
# 特征工程（在工程师提供的120+特征基础上）
- 新增特征：加购物车未支付次数、浏览深度变化率
- 时间窗口：用T-30到T-8天的行为预测T日是否流失

# 模型选择：
- 尝试LightGBM、XGBoost、逻辑回归
- 最终选LightGBM（AUC=0.87, 召回率78%）

# 可解释性分析：
- SHAP值揭示：客服投诉未解决是流失最强信号
```

**交付物**：
- 模型文件 `churn_model_v1.pkl`
- 特征重要性报告（给业务团队）
- 模型卡片（记录数据版本、性能指标、局限性）

---

### **Phase 5: 模型生产化部署（Week 8-9）**
**ML工程师**

**挑战**：科学家的Jupyter Notebook无法直接给10万用户实时打分

**实现方案**：
```
架构设计：
┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│ Airflow每日  │───>│ 特征服务      │───>│ 模型API      │
│ 触发特征计算 │    │（Feast/Redis）│    │（FastAPI）   │
└─────────────┘    └──────────────┘    └─────────────┘
                                             │
                                             v
                                    ┌─────────────────┐
                                    │ CRM系统消费预测  │
                                    │ 结果触发营销动作 │
                                    └─────────────────┘
```

**关键决策**：
- 用Redis缓存特征（而非每次实时计算）：延迟从500ms降至30ms
- 模型用ONNX格式部署（而非pickle）：推理速度提升3倍
- 建立A/B测试框架：10%流量验证新模型再全量

**监控体系**：
- 数据漂移监控：每日检测特征分布变化（PSI指标）
- 模型性能监控：线上实际AUC vs 离线验证AUC
- 业务指标监控：干预用户的次日留存率

## 角色协作的关键接口

| 交接环节 | 关键产出物 | 常见摩擦点 |
|---------|-----------|-----------|
| 工程师→分析师 | 数据字典、特征表 | 特征定义不清（"活跃用户"口径不统一）|
| 分析师→BI | SQL查询逻辑、指标定义 | 分析师ad-hoc查询性能差，BI需重构 |
| 分析师→科学家 | 业务假设、标注数据 | 样本偏差（分析师只给了投诉用户样本）|
| 科学家→工程师 | 模型文件、依赖清单 | Python版本不兼容、缺少预处理代码 |

---

## 这个流程的演进路径

**成熟度Level 1**（大多数公司现状）：  
工程师建仓库 → 分析师写SQL → 业务方看Excel

**成熟度Level 2**（本案例）：  
+ BI自动化报表 + 预测模型辅助决策

**成熟度Level 3**（AI Native）：  
闭环自动化：模型预测→自动触发营销动作→效果回流再训练模型

---

**一句话总结**：数据架构师定规则，工程师修路铺管道，分析师开车探路，BI开发者修建公交系统，科学家发明自动驾驶，ML工程师让自动驾驶上路运营。

### 🔷 高分 vs 低分表达对比（4–6组）

**直击中国考生最常见的“低分替换”习惯。**

| ❌ 低分写法                                  | ✅ 高分替换                                                                         | 说明（关联笔记内容）                      |
| --------------------------------------- | ------------------------------------------------------------------------------ | ------------------------------- |
| _try to fix problems after they happen_ | _**shift from passive response to proactive intervention**_                    | 笔记核心概念，描述现代工作模式的转变，有深度          |
| _check things sometimes_                | _**deploy real-time monitoring systems**_                                      | 将“检查”具体化为“部署实时监控系统”，体现技术思维      |
| _make things work better now_           | _**lay the foundation for scalable automation**_                               | 笔记中“成熟度Level 3”的体现，着眼于未来发展      |
| _work together closely_                 | _**collaborate across key interfaces (e.g., between engineers and analysts)**_ | 笔记中“角色协作”部分的精准提炼，更学术            |
| _use data to make decisions_            | _**support data-driven decision-making**_                                      | 固定搭配，是数据分析领域的标准术语，比“use data”地道 |
