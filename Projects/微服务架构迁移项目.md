# 微服务架构迁移项目

## 项目概述

将公司核心产品从单体架构（monolithic Ruby on Rails 应用，500K+ LOC）迁移至微服务架构，以提升系统可扩展性、团队开发效率和部署灵活性。

**项目背景**:

- **现状**: 单体应用部署周期 2-3 天，任何代码变更都需要全量重新部署
- **瓶颈**: 数据库连接池耗尽（2000+ 连接）、测试时间长（2h+）、新功能上线需要全团队协调
- **业务压力**: 用户增长 3x/年，黑五期间响应时间 P99 > 5s

**问题陈述**:
> "当前单体架构已成为业务增长瓶颈。我们需要一个架构，让团队独立开发、快速部署，同时保证系统稳定性。"

**机会/价值**:

- **开发效率**: 团队独立部署，从 2-3 天 → < 30 分钟
- **系统可靠性**: 单个服务故障不影响全局（可用性目标 99.9% → 99.95%）
- **技术栈现代化**: 从 Ruby 2.7 → 多语言（Go, Python, Node.js），可选择最适合的工具

## 目标与范围

### 项目目标

**战略目标**: 构建可支撑未来 5 年增长（10x 用户）的技术架构

**SMART 目标**:

1. **S**pecific: 将单体应用拆分为 15 个核心微服务
2. **M**easurable:
   - 部署频率: 从每周 1 次 → 每天 10 次
   - 变更前置时间: 从 2-3 天 → < 1 小时
   - 服务恢复时间: 从 2-4 小时 → < 15 分钟
3. **A**chievable:
   - 分 4 个阶段迁移，每个阶段 2-3 个月
   - 优先迁移非核心服务（如通知、搜索），降低风险
4. **R**elevant: 支持业务快速迭代，新产品线并行开发
5. **T**ime-bound: 9 个月完成（2025-07-01 → 2026-03-31）

### 范围界定

**包含范围内** (In Scope):

- 核心业务服务拆分（用户、订单、支付、商品、库存、搜索、通知等 15 个服务）
- API Gateway 实现（Kong）
- 服务间通信（gRPC + 消息队列 Kafka）
- 数据库分离（PostgreSQL 按服务拆分）
- 可观测性（Prometheus + Grafana + ELK）
- CI/CD 流水线（GitLab CI + ArgoCD）

**明确排除** (Out of Scope):

- 前端重构（保持现有 SPA 架构）
- 数据中心迁移（继续使用 AWS）
- 第三方集成重构（仅在必要时适配）

## 决策记录

### 2025-06-15: 选择 gRPC 而非 REST

**背景**: 需要选择服务间通信协议

**决策**: 使用 gRPC (Protocol Buffers) 而非 REST (JSON)

**理由**:

- 性能更好（消息大小小 40%+，序列化快 5-10x）
- 强类型（代码生成，减少运行时错误）
- 支持双向流（实时通信场景）

**替代方案**:

- REST: 简单但性能差
- GraphQL: 灵活但增加复杂度

**影响**: 开发效率初期受影响（学习 Protobuf），但长期收益明显

### 2025-07-20: 选择 Saga 模式而非 2PC

**背景**: 需要解决分布式事务问题

**决策**: 使用 Saga 模式（补偿事务）而非两阶段提交（2PC）

**理由**:

- 2PC 性能差（锁定资源，可用性低）
- Saga 更灵活（最终一致性，适合业务场景）
- 可追溯（每个步骤有日志）

**替代方案**:

- 2PC: 强一致性但性能差
- TCC (Try-Confirm-Cancel): 更复杂，收益不明显

**影响**: 业务逻辑复杂度增加，但系统整体性能和可用性提升

### 2025-10-10: 选择 PostgreSQL 而非 MongoDB

**背景**: 商品服务需要文档数据库还是关系型数据库？

**决策**: 继续使用 PostgreSQL（JSONB 支持）而非引入 MongoDB

**理由**:

- 减少技术栈复杂度（团队熟悉 PostgreSQL）
- PostgreSQL 9.5+ 的 JSONB 性能满足需求
- 避免多数据库运维成本

**替代方案**:

- MongoDB: 灵活但增加运维成本
- MySQL: JSON 支持不如 PostgreSQL

**影响**: 技术栈统一，运维效率提升

## 项目回顾

### 启动后评估

**实际 vs 目标**:

| 指标 | 目标 | 实际 | 达成率 |
|------|------|------|--------|
| 服务数量 | 15 | 15 | 100% |
| 部署频率 (次/周) | 10 | 12 | 120% |
| 变更前置时间 | < 1h | 0.5h | 200% |
| 系统可用性 | 99.95% | 99.96% | 100.01% |
| 预算 | $500K | $520K | 96% |
| 工期 | 9 个月 | 8.5 个月 | 106% |

**成功因素**:

- **分阶段迁移**: 从低风险服务开始，降低风险，团队积累经验
- **基础设施先行**: CI/CD、监控、日志系统提前就绪，支持快速迭代
- **自动化测试**: 单元测试覆盖率 85%+, 集成测试自动化，减少回归问题
- **团队培训**: 定期技术分享、结对编程，快速提升团队技能

**挑战**:

- **分布式复杂性**: 服务间通信、数据一致性、故障排查难度增加
- **学习曲线**: 团队需要掌握 Kubernetes、Istio、gRPC 等新技术
- **运维成本**: 微服务数量增加，监控、告警、日志管理复杂度提升

### 经验总结

**可复用实践**:

1. **Strangler Fig 模式**: 逐步替换单体应用，而非大爆炸式重写
2. **API 优先设计**: 先定义 API 契约，再实现逻辑，便于并行开发
3. **金丝雀发布**: 新版本先小流量验证，再逐步推广，降低风险
4. **自动化一切**: CI/CD、测试、部署、监控，减少人工错误

**避免错误**:

1. **不要过早拆分**: 初期只有 3-5 个服务足够，避免过度拆分
2. **不要忽视可观测性**: 监控、日志、追踪是微服务的生命线
3. **不要共享数据库**: 每个服务独立数据库，避免耦合
4. **不要忘记文化变革**: 微服务不仅是技术变革，也是组织变革（需要 DevOps 文化）

**可复用到其他项目**:

- **迁移方法论**: 分阶段迁移、双写验证、灰度发布的通用流程
- **技术选型框架**: 评估微服务技术栈的决策矩阵
- **培训计划**: 微服务技术栈的学习路径和资源清单
