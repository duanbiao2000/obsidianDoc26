# WebSocket 实时通信功能实现

## 概述

为 ChatApp 添加 WebSocket 支持，实现用户间实时消息推送、在线状态同步、多设备消息同步。替代当前的长轮询方案（响应时间 2-5s → 目标 < 100ms）。

**背景**: 当前基于 HTTP 长轮询的实时性不足，用户反馈"消息延迟明显"，影响产品竞争力（竞品 Slack、Discord 均已采用 WebSocket）。

**为什么重要**:

- **用户体验**: 消息延迟从秒级降至毫秒级
- **技术债务**: 长轮询方案在高并发下资源浪费严重（每次请求都携带完整 HTTP header）
- **扩展性**: WebSocket 为未来功能（如实时协作编辑、语音通话）打基础

## 目标与交付物

### 主要目标

构建高可用的 WebSocket 服务，支持 10K 并发连接，消息延迟 < 100ms (P99)。

**成功标准**:

- [x] 消息端到端延迟 < 100ms (P99)
- [x] 支持 10K 并发连接（单实例）
- [x] 连接断开后自动重连，消息不丢失
- [x] 多设备在线状态同步 < 500ms
- [x] 99.9% 消息送达率

## 阻塞与问题

### 历史阻塞

**阻塞 1: Redis 集群 Pub/Sub 不持久化** (2026-02-18)

- **解决方案**: 使用 Redis Streams 存储离线消息， Pub/Sub 仅用于实时推送
- **解除时间**: 2026-02-19

**阻塞 2: Nginx WebSocket 连接频繁断开** (2026-02-22)

- **原因**: Nginx 默认 timeout 过短（60s），WebSocket 长连接被主动断开
- **解决方案**: 增加 `proxy_read_timeout`, `proxy_send_timeout` 至 3600s
- **解除时间**: 2026-02-23

### 已解决问题

| 问题 | 解决方案 | 影响 |
|------|----------|------|
| 内存泄漏（10K 连接） | 避免闭包引用 | 无 |
| 消息重复（多设备） | 客户端消息去重（messageId） | 无 |
| 证书验证失败 | 配置 SSL context | 无 |

### 技术决策记录

**决策 1: 使用 `ws` 而非 `Socket.io`**

- **理由**: 性能更好（消息吞吐高 40%）、依赖更少、生态更广泛
- **权衡**: 放弃自动 fallback（可以接受，目标用户主要在 Chrome/Firefox）

**决策 2: JSON 而非 MessagePack**

- **理由**: 可读性强（调试友好）、序列化性能差异可接受（< 1ms）
- **权衡**: 消息大小略大（~30%），但对 < 1KB 消息影响不大

**决策 3: Redis Streams + Pub/Sub 组合**

- **理由**: Pub/Sub 实时推送 + Streams 持久化离线消息
- **权衡**: 引入额外复杂度，但满足消息不丢失需求
